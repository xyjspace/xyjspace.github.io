引出SqlSession

SqlSession是什么？

官方文档对SqlSession的解释为：

The SqlSession contains absolutely every method needed to execute SQL commands against the database. You can execute mapped SQL statements directly against the SqlSession instance.
翻译过来就是SqlSession是我们与mybatis进行交互的API，通过SqlSession我们可以对数据库进行增删改查操作。

到现在为止整个过程已经连成了一条线有木有。

我在这里总结一下截止到目前的整个流程。

上面的是逆向查找，主要是记录了我在整个分析过程中的思路。下面正向总结一下：

  1. SpringBootStarter启动，执行@Configuration注解的处理器ConfigurationClassPostProcessor
  2. 因为ShardingTestStarter（我项目的启动类）拥有@SpringBootApplication注解，同时该注解上有@Configuration注解，所以该类会被ConfigurationClassPostProcessor处理器处理。在ShardingTestStarter类上我们加了@MapperScan注解，该注解中又加入了一个MapperScannerRegistrar.class的bean.所以又触发了ImportBeanDefinitionRegistrar
  3. 执行ImportBeanDefinitionRegistrar的registerBeanDefinitions方法将我们指定的包下面的类BeanClass指定为MapperFactoryBean
  4. 在Spring初始化bean的时候，会为MapperFactoryBean设置SqlSessionFactory这个成员变量。而在设置这个成员变量的时候会去生成SqlSessionTemplate对象。
  5. 因为MapperFactoryBean继承了FactoryBean，因此在Spring中还会执行.getObject方法（执行时间不明，以后再分析）
  6. 在执行MapperFactoryBean.getObject()方法的时候，调用sqlSessionTemplate.getMapper()方法
  7. 调用sqlSessionTemplate.getMapper()方法的时候调用Configuration.getMapper(sqlSessionTemplate)方法
  8. 调用mapperRegistry.getMapper(sqlSessionTemplate)方法
  9. 执行mapperProxyFactory.newInstance(sqlSessionTemplate)方法，并将sqlSessionTemplate作为参数传去
  10. 得到带有sqlSessionTemplate对象的MapperProxy，并注入到bean
  11. 项目部署成功
  12. 项目启动后执行userMapper.insert()方法的时候，通过MapperProxy进行调用。
  13. 拿到 MappedStatement对象。
  14. 执行sqsession.insert(mappedStatement, paramater)方法

executor是什么？

我们上边有讲，SqlSession是用户操作Mybatis的窗口，是Mybatis对外暴露的API。那么Executor就是Mybatis直接操作Mapper的工具。通过Executor Mybatis就可以去执行各种方法，比如insert,或者update. 并且Executor是和SqlSession绑定在一起，每个SqlSession拥有一个Executor变量。该变量由Configuration生成。

我们继续：
接下来的步骤为：

  1. CachingExecutor查看是否需要刷新缓存。这里刷新的是MappedStatement的缓存。对应的就是Mybatis的二级缓存，默认为关闭。
  2. 因为这里执行的是插入操作，二级缓存并不起作用，直接使用SimpleExecutor执行update()方法。
  3. 执行SimpleExecutor父类BaseExecutor中的update方法，然后调用doUpdate()方法。该方法为抽象方法，由子类实现。（模板方法模式）
  4. 生成StatementHandler，这个方法由Configuration通过RoutingStatementHandler类生成，根据MappedStatement中的statementType来决定是生成普通的StatementHandler, PrepareStatementHandler,还是CallAbleStatementHandler. 我们这里生成的是PrepareStatementHandler. 过程中还会通过MappedStatement和parameters生成sql语句
  5. 用InterceptorChain去处理statementHandler.
  6. 根据Executor中的transaction拿到Connection();
  7. 执行PrepareHandler.prepare()方法，执行instantiateStatement()方法.
  8. 注意注意前方高能！！！ 来到核心的方法了： 这里会生成我们最终想要的类 ShardingPrepareStatement
